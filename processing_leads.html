Universal Planning Cleaner

A self-contained app that pulls together messy CSV data from four different systems, untangles the IDs, joins everything to the right centre, and produces a single prioritised call sheet.

01What it does

Takes a list of leads in a territory and enriches every row with its full visit history, purchase record, and a priority tag the reps use to decide who to call.

The input is a masterlist, the authoritative list of every centre in a territory. The job is to join that list against several other files: visit logs from the field, accounting exports from MYOB, and other masterlists that act as the source of truth for centre IDs. The output is one CSV where every lead has been annotated with everything the rep needs to know about it.

The work that's hard to see is in the joining. Centre IDs in the source data are unreliable, the same physical centre can appear under three different IDs across two visit logs and a masterlist. Centre names drift as places get renamed or rebranded. Apostrophes, hyphens and suburb suffixes don't match across systems. A lot of the code exists to make the join actually correct.

Why this exists The data lives across four upstream systems with no shared key. Every UID is hand-typed by reps in the field, so the same centre appears under different IDs in different files. Reconciling this by hand is what this app replaces.

02How it's built

Pure Python, no installs needed. The web server, file parser, CSV reader and job runner are all built from the standard library so anyone can run it without setting up dependencies.

Web layer

Built-in http.server with one request handler. The UI is served from index.html; everything else goes through JSON endpoints under /api/.

File upload parser

A custom multipart parser, because the standard library version was removed in Python 3.13. It splits the upload on the boundary marker and returns a clean dict of fields and files.

Background jobs

Each job runs in its own thread. A shared dictionary (with a lock) holds the live state, so the UI can poll /api/jobs/<id> for log lines and progress.

Storage

One config.json stores the registered references. Outputs land in a fixed directory outside the app folder so they survive restarts. Modified masterlists are backed up before any change is written.

03Reference files

Every file registered with the app is tagged with a type. The pipeline uses the type to decide what role each file plays.

TypeWhat it provides
masterlist The source of truth for centre IDs. Used to fix wrong IDs everywhere else, and as the universe of "all centres in this territory."
archive The historical visit log. One row per rep visit, with the result, comment, and notes.
transaction_log Same shape as archive, used for more recent visits. Treated identically by the pipeline.
myob The accounting export. Tells us which centres have actually paid for something, and what they bought.

04The three jobs

The same upload form runs one of three jobs depending on what the user picks.

Enrich a call sheet

Takes a masterlist, pulls in every visit and every MYOB book on file, computes a priority, and writes one enriched row per lead. This is what reps work from day to day.

Find unassigned leads

Takes the combined sheet of every rep's current leads, compares it against the full masterlist, and produces a CSV of every lead that nobody is currently working. The enrichment is the same as above — the difference is only in which leads are included.

Update masterlist names

The opposite direction. Takes a leadsheet, looks up each centre by UID in the masterlist, and writes the leadsheet's name back into the masterlist whenever it differs. The original masterlist is backed up first, and a CSV report of every change is written.

05Fixing broken IDs

The hardest part of the pipeline. Centre IDs are unreliable in three ways: typos, drift after a centre changes name, and outright theft (a rep accidentally pastes another centre's UID into the wrong row).

Two-way matching

From the masterlist references, we build two lookups:

  • name → correct UID
  • UID → centre name

Forward match. For every row, normalise the centre name and look up the correct UID. If it differs from the UID on the row, replace it and log the change.

Reverse check. If the name isn't in the reference at all, take the UID from the row and look up which centre it actually belongs to. If that centre has a different name, the UID was stolen — strip it from the row and log it.

Subtlety Reference names often have the suburb appended ("Sunny Days Tarragindi") while the visit log just has the bare name ("Sunny Days"). A naive comparison would flag every one of these as theft. The check ignores the difference if either name is a prefix of the other.

Why two normalisers

Different upstream systems handle apostrophes and hyphens differently, so the same name has to be normalised two different ways:

# For ID lookup apostrophes & hyphens become spaces
def normalize_for_id(name):
    s = re.sub(r"['‘’\-]", ' ', name)
    s = re.sub(r'\s+', ' ', s)
    return s.lower().strip()

# For MYOB lookup, apostrophes are removed entirely
def normalize_for_myob(name):
    return re.sub(r"['‘’ʼ]", '', name).lower().strip()

MYOB exports strip apostrophes ("O'Hara's" → "oharas"), so the MYOB normaliser has to do the same. The ID normaliser keeps a space in the same position, so "O'Hara", "O Hara", and "O-Hara" all collide on lookup.

Side reports

Every run that touches IDs writes two extra CSVs alongside the main output:

  • id_corrections, every change made (where, what was, what is now).
  • id_unmatched, every centre that couldn't be matched to any reference.

06Removing duplicates

"Duplicate" means different things depending on whether the row carries notes.

key = ('notes', uid, notes)  if notes else ('blank', uid, visit)

If a row has notes, the notes themselves are what makes the row distinct, two visits to the same centre on the same day with different notes are two different events. If a row has no notes, the visit date is what makes the row distinct. We don't want three blank copies of the same visit just because it appears in three log files.

The masterlist gets its own dedup pass after IDs are corrected, keeping the first row for each UID. This matters because once duplicate IDs are unified to a single canonical UID, the rows themselves become exact duplicates.

07Counting visits

For each centre, roll up the deduplicated visit history into the numbers that drive the priority decision.

A row only counts as a "visit" if all of these are true:

  1. The UID belongs to a centre on the masterlist.
  2. The result is one of the in-person outcomes, Successful sale, No sale, or DM absent on initial visit. Phone calls and emails are not counted as visits.
  3. The visit date can be parsed (six date formats are accepted).

For every centre, the rollup produces:

FieldWhat it means
Successful sales countHow many visits ended in a sale
No-sale countHow many visits ended without a sale
Last sale dateThe most recent successful visit
Last no-sale dateThe most recent unsuccessful visit
Visit historyEvery visit, sorted newest first, with date, result and notes
Last commentThe most recent comment from any touchpoint, phone and email comments are kept here even though they don't count as visits

If the same centre is visited on the same day in two different log files, it counts once.

08Matching to MYOB

A name-based join between the masterlist and the accounting system. The matcher has to bridge two systems that store the same centre under cosmetically different names, different punctuation, different case, sometimes the address blob jammed into the same field.

The accounting system has no shared key with the masterlist, there's no UID column to join on, just the customer name. And the customer name in MYOB is whatever the bookkeeper typed in: "OHaras" in MYOB might be "O'Hara's Early Learning" in the masterlist. The matcher has to make those collide.

What a MYOB export looks like

The matcher reads two columns from each MYOB CSV. Everything else is ignored.

ColumnIndexWhat it holds
Customer3The centre name as the bookkeeper typed it
Book4One line item, the title of a book the centre purchased

One centre usually appears across many rows (one per book purchased). The matcher accumulates them all, then deduplicates within each centre while preserving the order they appeared.

The normalization pipeline

Every centre name, both sides of the join, runs through the same three-step normalizer before it's used as a lookup key:

def normalize_for_myob(name):
    return re.sub(r"['‘’ʼ]", '', name).lower().strip()
  1. Strip apostrophes. Four characters get removed: the straight ASCII apostrophe ('), the curly left and right single quotes (' '), and the modifier letter apostrophe (ʼ). MYOB stores names without apostrophes; the masterlist keeps them. Removing all four variants on both sides makes them collide.
  2. lowercase everything. trivial but essential, the same name capitalised differently in two systems would otherwise miss.
  3. Strip surrounding whitespace. Catches stray leading or trailing spaces from sloppy data entry or from the Excel export pipeline.

Building the lookup index

The MYOB side is loaded into a dictionary keyed by the normalised customer name, with the value being the list of every book that customer has bought:

myob_books = defaultdict(list)

for row in myob_csv:
    cust = normalize_for_myob(row[3])   # customer column
    book = row[4].strip()              # book line item
    if cust and book:
        myob_books[cust].append(book)

# Deduplicate within each customer, preserving order:
for k in myob_books:
    myob_books[k] = list(dict.fromkeys(myob_books[k]))

The dict.fromkeys trick is how Python deduplicates a list while preserving insertion order, newer Python guarantees dict ordering, so the first occurrence of each book stays first.

The masterlist side: extracting the centre name

The masterlist's Details column is messy, it often holds a multi-line blob with the centre name on the first line, then address, phone, and contact details on subsequent lines:

Sunny Days Early Learning
123 Stanley St
Tarragindi QLD 4121
07 3892 0000

Before the row can be matched against MYOB, the centre name has to be pulled out of that blob. A small helper does it:

def extract_centre_name(cell):
    return cell.split('\n')[0].strip()

Split on newline, take the first line, strip whitespace. The result then goes through the same normalize_for_myob as the MYOB side, and the keys are now directly comparable.

The match itself

Once both sides are normalised, the actual lookup is a single dictionary access, no fuzzy matching, no Levenshtein distance, just a hash hit or miss:

centre_name = normalize_for_myob(extract_centre_name(details))
books = myob_books.get(centre_name, [])

if books:
    books_str = '\n'.join(f'• {b}' for b in books)
    book_count = len(books)
    matched_myob += 1
else:
    books_str = 'needs manual verification'
    book_count = ''

A hit writes a bullet list of every book the centre has bought into the Books Purchased (MYOB) output column, and the count into # Books Purchased. A miss writes the literal string needs manual verification — that's the signal to ops that this row needs a human-eye match (the centre might genuinely not be a customer, or the names may differ in a way the normaliser can't bridge).

The DEAD → BOTTOM override

The MYOB result also feeds back into the priority logic. After the priority tag is computed from the visit history, one final check runs:

if priority == 'DEAD' and books:
    priority = 'BOTTOM'
Why this override exists A centre flagged DEAD by the visit log alone (six or more no-sales, never a yes) gets bumped down to BOTTOM if MYOB shows them as a paying customer. A "no sale" history doesn't always mean the centre never bought — the sale might have happened outside the rep flow (an online order) or it just wasn't logged against the lead at the time. MYOB is the system of record for what was actually purchased, so if it shows books on file the centre is a real customer and shouldn't be marked dead.

09The priority rules

Each lead gets one tag based on three things: how many times it's been visited, whether it's ever bought, and how recently. The same logic applies to both Australia and New Zealand.

SituationTag
This centre has never been visited by a rep NEW
The most recent visit ended in a sale BEST
Has bought before, and the last sale was 2–3 visits ago HIGH
Has bought before, but the last sale was 4–5 visits ago MEDIUM
Visited 1-3 times, never bought, still worth chasing POTENTIAL
Visited 4-5 times, never bought, odds are getting low LOW
Has bought before, but the last sale was a long time ago BOTTOM
Visited 6+ times, never bought, give up unless MYOB says otherwise DEAD

The MYOB override

Applied after the rule lookup: if the tag came out as DEAD but MYOB shows books on file, the tag is downgraded to BOTTOM. This is the only cross-system override in the priority logic.

11Tricky cases

Each item below is a real problem in the source data with a deliberate fix in the code.

  • Missing header row. If the first row of a masterlist looks like data (its first cell is a UID), a synthetic header is added and the row is treated as data.
  • Junk columns past column L. Input is hard-capped at the first 12 columns; anything beyond is dropped before processing so output rows stay clean.
  • Excel BOM characters. All readers use utf-8-sig so files exported from Excel don't poison the first column header.
  • Six date formats. The date parser tries six common formats in order. A row with an unparseable date still goes through the pipeline — it just doesn't contribute to date-based aggregates.
  • Stolen IDs that are actually fine. When the reference name is the leadsheet name plus a suburb, the prefix-match guard prevents a false theft alarm.
  • Manually-touched leads in the unassigned pass. Leads with a non-empty Contact Status are excluded from the unassigned check, so anything a rep has already touched isn't re-surfaced.
  • Backups before any masterlist write. The "update names" job copies every modified masterlist into backups/ with a timestamp before writing, so a bad update can be rolled back.
  • Local file or upload. References can be registered either by upload (the file gets copied into the app's data folder) or by absolute local path (the file is read in place). Both modes share the same processing code.
  • Trimmed columns in the unassigned output. Five legacy columns are dropped from the unassigned CSV (Last Visit Notes, Last Location Update, etc.) to keep the call sheet narrow.
