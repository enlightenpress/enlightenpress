What is the Odometer Tracker?

A live dashboard that automatically pulls odometer readings from field reps and turns them into clear, week-by-week kilometre reports.

speed

What it tracks

Each sales rep's vehicle odometer reading at the start and end of every working day, giving an accurate picture of how far they drove.

calculate

What it calculates

The daily kilometres driven (end reading minus start reading), plus weekly totals and a daily average — all calculated automatically.

photo_camera

Photo verification

Reps submit a photo of the odometer at check-in and check-out. Managers can click to view the actual image at any time.

people

Multiple reps

View a single rep or all reps at once, each with their own complete report card — side by side for easy comparison.

How the System Works

The tracker connects three pieces together: the reps' data submissions, a Google Sheet that stores everything, and this dashboard that reads and displays it beautifully.

→
🖥️

lightbulb

In plain English: Think of it like a digital logbook. Every morning a rep records their odometer reading when they start driving, and every evening when they finish. This tool reads all those records and instantly shows you a clean report of how far everyone drove — by day, by week, or across any date range you choose.

account_tree Process

Data Flow — Step by Step

Here is exactly what happens from when you open the dashboard to when you see a completed report.

  1. Dashboard loads

    When you open the web app, it immediately contacts the Google Apps Script server and calls the getDashboardData() function. A spinning loader is shown while this is happening.

  2. Server reads Google Sheets

    The server reads every rep's sheet — each rep has their own tab in the Google Sheet. The data is read and sent back to the dashboard as a structured package. To avoid reading the sheet every single time, the server caches the data (stores a temporary copy). This makes subsequent loads much faster.

  3. Filters are set up

    The dashboard populates the rep dropdown with all known rep names, sorted alphabetically. The date range is automatically set to the current Monday–Friday working week so you see today's data immediately.

  4. Report is generated

    The dashboard matches each row of data to a date, pairs up Check In and Check Out records, subtracts start from end odometer to get daily KM, then groups the days into Monday–Friday weeks before rendering the cards you see on screen.

  5. Navigation is set up

    After the report renders, the sidebar (desktop) and mobile pills are built out so you can jump between reps instantly. A scroll-aware sticky header also activates so you always know which rep's data you are looking at.

Feature Overview

A summary of everything the Odometer Tracker can do.

  • filter_alt
    Flexible Date Range Filtering
    Pick any start and end date to view historical data, run a custom period report, or focus on a specific week. The dashboard defaults to the current Mon–Fri for instant daily visibility.
  • person
    Individual or All-Reps View
    Use the rep selector to view a single rep in isolation, or choose "All Reps" to generate a complete report for everyone at once — each rep gets their own clearly labelled section.
  • calendar_today
    Automatic Week Grouping
    Days are automatically grouped into Monday–Friday working weeks. Each week shows a header with running totals and an average, plus a summary bar at the bottom.
  • calculate
    Automatic KM Calculation
    The system calculates daily KM automatically: it takes the end-of-day odometer reading and subtracts the start-of-day reading. No manual arithmetic required.
  • photo_camera
    Photo Verification
    Every odometer reading can be backed by a photo. A camera icon appears next to any reading where a photo was submitted. Clicking it opens the image in full screen in a new browser tab, loading it directly from Google Drive.
  • sidebar
    Rep Sidebar (Desktop)
    On larger screens, a fixed sidebar on the left lists all reps in the current report. Clicking a name scrolls directly to that rep's section. The currently visible rep is automatically highlighted as you scroll.
  • smartphone
    Mobile Navigation Pills
    On phones and tablets, a horizontal scrollable row of rep name pills appears below the filters. Tapping a pill jumps to that rep's data.
  • push_pin
    Sticky Rep Header
    As you scroll down through a long report, a sticky banner slides in at the top of the screen showing the name of the rep whose data is currently in view. This prevents losing track of which rep you are looking at.
  • cached
    Data Refresh
    The "Refresh Data" button clears the server cache and re-reads the Google Sheet from scratch. Use this to pull in new submissions that were added after the dashboard was initially loaded.
  • devices
    Fully Responsive Design
    The layout adapts to any screen size. Day cards reflow from a 3-column grid on desktop down to a single column on mobile. All headers and summaries also restructure gracefully.

At the top of the dashboard is a filter bar with four controls. Here is what each one does.

Control What it does Default
Select Rep Choose a single rep by name, or select "All Reps" to see everyone at once. All Reps
Start Date The first day of the date range you want to report on. Monday of current week
End Date The last day of the date range. The system includes all data up to 11:59 PM on this date. Friday of current week

label Status Indicators

Status Badges

Every day card has a coloured badge in the top-right corner that instantly tells you the state of that day's data.

Recorded
Both a start and end odometer reading are present. Daily KM has been successfully calculated. This is the desired state.
Incomplete
Only one reading is present — either a Check In or a Check Out was submitted, but not both. Daily KM cannot be calculated. Action may be required.
No Data
No odometer data was submitted for this day. The day falls within the date range but no form submission exists for it.
check_circle

At a glance: A healthy week shows all five day cards with a green "Recorded" badge. Yellow badges indicate follow-up is needed with the rep to complete their submission.

table_chart Data Structure

Data Structure (Google Sheets)

The dashboard reads data from a Google Sheet. Each sales rep has their own sheet tab. Here are the columns the system looks for:

Column Name Type Description Required?
WorkDate Date The date the rep worked. Accepted formats: YYYY-MM-DD or DD/MM/YYYY. Required
Status Text Must contain either "check in" or "check out" (case-insensitive). This determines whether the row is treated as a start-of-day or end-of-day entry. Required
Start of Day Odometer Number Number The odometer reading at the beginning of the day, in kilometres. Read from Check In rows only. Conditional
End of Day Odometer Number Number The odometer reading at end of day, in kilometres. Read from Check Out rows only. Conditional
Start of Day Odometer Picture Text / URL A Google Drive file path or identifier for the start-of-day odometer photo. Used to load the image when the camera icon is clicked. Optional
End of Day Odometer Picture Text / URL A Google Drive file path or identifier for the end-of-day odometer photo. Optional
info

How pairing works: For each date, the system looks for a Check In row (to get the start reading) and a Check Out row (to get the end reading). If both are found, daily KM is calculated. If only one is found, the day is marked "Incomplete". The column headers must match exactly as listed above.

Refreshing Data

The dashboard uses a server-side cache to load quickly. Here's what that means and when to use the refresh button.

bolt

Why caching exists

Reading a large Google Sheet every time someone opens the dashboard would be slow. The server saves a copy of the data so repeat visits load almost instantly.

cached

When to refresh

Click "Refresh Data" whenever a rep has submitted new data since you first opened the dashboard, or if you suspect the data shown is out of date.

check_circle

What refresh does

It calls clearDashboardCache() on the server to delete the stored copy, then immediately re-reads all sheets fresh. The report re-generates automatically.

photo_camera Photo Viewer

Viewing Odometer Photos

Every odometer reading can be backed by a physical photo of the dashboard — providing an auditable trail and discouraging inaccurate data entry.

  1. Spot the camera icon

    On a day card, if a photo was submitted, a small camera icon (📷) appears next to the odometer reading. Start readings show a teal icon; end readings show a coral icon.

  2. Click to open

    Clicking the icon immediately opens a new browser tab. A "Loading image..." message is shown while the image is being fetched from Google Drive. This two-step approach is needed to satisfy browser popup-blocker rules.

  3. Image loads full-screen

    The Apps Script server retrieves the image from Google Drive, converts it to a data URL, and sends it back to the tab. The image is then displayed centred on a black background for easy reading.

warning

If no camera icon appears on a reading, it means no photo was submitted for that entry. The reading is still valid, but there is no photo evidence to view.

Technical Architecture

For developers or technical stakeholders — a summary of the technology stack and design decisions.

Layer Technology Role
Frontend HTML / Vanilla JS Single-page dashboard. No framework dependencies beyond Tailwind CSS (loaded via CDN) and Material Icons.
Styling Tailwind CSS Utility-first CSS framework for rapid, consistent layout. Custom CSS handles dynamic HTML that Tailwind's JIT compiler cannot purge-scan.
Backend Google Apps Script Serverless JavaScript runtime hosted on Google's infrastructure. Exposes getDashboardData(), clearDashboardCache(), and getOdometerImage() as callable server functions.
Data Store Google Sheets One tab per sales rep. The backend reads rows using the Sheets API via Apps Script's built-in SpreadsheetApp service.
File Storage Google Drive Odometer photos are stored in Google Drive. The backend retrieves them using DriveApp, converts them to base64 data URLs, and streams them to the browser.
Caching Apps Script Cache Service Server-side caching reduces Google Sheets API calls. Cache is invalidated manually via the Refresh button.
Date handling Custom JS Date strings are parsed to local-time Date objects (avoiding UTC offset issues). Two formats are supported: YYYY-MM-DD and DD/MM/YYYY.
Scroll detection IntersectionObserver API Used for both the sticky rep header and sidebar active-state highlighting. No polling — purely event-driven.
Security HTML escaping All user-supplied and sheet-sourced text is passed through an escapeHtml() function before being injected into the DOM, preventing XSS attacks.
build

Server functions called by the dashboard: getDashboardData() — returns all rep data as a structured JSON object; clearDashboardCache() — purges the server-side cache; getOdometerImage(imagePath) — fetches an image from Drive and returns it as a base64 data URL string.
